üîπ Dependency Injection in .NET Core ‚Äì Interview Q&A

1. What is Dependency Injection (DI)?
Dependency Injection is a design pattern where a class‚Äôs dependencies are provided (injected) from the outside instead of creating them inside the class. In .NET Core, DI is built-in and encourages loose coupling, testability, and maintainability.

2. What problem does DI solve?
‚úî Removes tight coupling between classes.
‚úî Makes code easier to test (mock dependencies).
‚úî Follows SOLID principles (especially Dependency Inversion Principle).

3. How is DI implemented in .NET Core?
In Program.cs (or Startup.cs in older versions), services are registered in the built-in IoC container using:
AddSingleton<T>(), AddScoped<T>(), AddTransient<T>()
Example:

builder.Services.AddScoped<IUserService, UserService>();

4. What are the DI lifetimes in .NET Core?
üîπ Transient ‚Üí New instance every time it‚Äôs requested.
üîπ Scoped ‚Üí One instance per HTTP request.
üîπ Singleton ‚Üí Same instance for the entire app lifetime.

5. Can you give an example of DI in a Controller?

public class OrderController : Controller
{
    private readonly IOrderService _orderService;
    public OrderController(IOrderService orderService) 
    {
        _orderService = orderService;
    }

    public IActionResult Index()
    {
        var orders = _orderService.GetAll();
        return View(orders);
    }
}

6. How do you inject services into non-controller classes?
‚úî Use constructor injection in custom classes.
‚úî Or inject IServiceProvider / IHttpContextAccessor.
Example:

public class ReportGenerator
{
    private readonly ILogger<ReportGenerator> _logger;
    public ReportGenerator(ILogger<ReportGenerator> logger)
    {
        _logger = logger;
    }
}

7. Can we inject multiple implementations of the same interface?
Yes ‚úÖ Register multiple and use IEnumerable<T>:

public ReportService(IEnumerable<IReportGenerator> generators) { ... }
Or use named options/factory pattern for conditional resolution.

8. What is the difference between AddSingleton, AddScoped, and AddTransient?
‚úî AddSingleton ‚Üí Same instance shared across all requests.
‚úî AddScoped ‚Üí Same instance per HTTP request.
‚úî AddTransient ‚Üí Always new instance per use.

9. How is DI different in .NET Core vs ASP.NET Framework?
üîπ ASP.NET Core ‚Üí DI is built-in.
üîπ Old ASP.NET Framework ‚Üí Needed third-party containers (Autofac, Ninject, Unity).

10. How can you resolve dependencies manually inside a method?
Inject IServiceProvider:

public class MyClass
{
    private readonly IServiceProvider _serviceProvider;
    public MyClass(IServiceProvider serviceProvider)
    {
        _serviceProvider = serviceProvider;
    }

    public void Run()
    {
        var myService = _serviceProvider.GetRequiredService<IMyService>();
    }
}

11. Can you inject configuration values?
Yes ‚úÖ using IOptions<T> / IOptionsSnapshot<T> / IOptionsMonitor<T>:

public class MyController
{
    private readonly MySettings _settings;
    public MyController(IOptions<MySettings> options)
    {
        _settings = options.Value;
    }
}

12. Can static classes be injected?
No üö´ because static classes cannot implement interfaces or have constructors. Instead, wrap them in a service and inject that.

13. What are some common pitfalls with DI?
‚ö† Registering wrong lifetime (e.g., Singleton DbContext ‚Üí threading issues).
‚ö† Overusing IServiceProvider (Service Locator anti-pattern).
‚ö† Over-injecting too many dependencies in one class.

    options.UseSqlServer(connectionString));

üîπ Middleware in .NET Core ‚Äì Interview Q&A

1. What is Middleware in .NET Core?
Middleware is a software component in the request pipeline that processes HTTP requests and responses. Each middleware can:
‚úî Do something before passing the request to the next middleware.
‚úî Do something after the next middleware finishes.

2. How does the Middleware pipeline work?
The pipeline is built in Program.cs (or Startup.cs in older versions).
Each middleware is executed in the order it is registered.
Example:

app.UseAuthentication();
app.UseAuthorization();
app.UseEndpoints(...);

3. What are some built-in Middlewares in ASP.NET Core?
‚úî UseRouting() ‚Üí matches incoming requests to routes.
‚úî UseAuthentication() ‚Üí handles authentication.
‚úî UseAuthorization() ‚Üí enforces access rules.
‚úî UseStaticFiles() ‚Üí serves CSS/JS/images.
‚úî UseExceptionHandler() / UseDeveloperExceptionPage() ‚Üí error handling.

4. How do you create custom Middleware in .NET Core?
Option 1: Using a class

public class CustomMiddleware
{
    private readonly RequestDelegate _next;
    public CustomMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {
        // before
        await context.Response.WriteAsync("Hello from middleware! ");
        await _next(context); // call next middleware
        // after
    }
}
Register:
app.UseMiddleware<CustomMiddleware>();
Option 2: Inline delegate
app.Use(async (context, next) =>
{
    await context.Response.WriteAsync("Before ");
    await next.Invoke();
    await context.Response.WriteAsync("After ");
});

5. What is the difference between Use, Run, and Map?
‚úî Use ‚Üí Adds middleware and calls the next one.
‚úî Run ‚Üí Terminates pipeline, does not call next.
‚úî Map ‚Üí Branches pipeline based on request path.
Example:

app.Use(async (ctx, next) => { await next(); }); // continue
app.Run(async ctx => { await ctx.Response.WriteAsync("End"); }); // stop
app.Map("/admin", appBuilder => { ... }); // branch

6. Can middleware be skipped or short-circuited?
Yes ‚úÖ If a middleware does not call await _next(context), the request pipeline stops there.

7. How do middleware and filters differ in .NET Core?
‚úî Middleware ‚Üí works at global request/response level, before MVC routing.
‚úî Filters ‚Üí specific to MVC/Web API actions (controller/action level).

8. What is the order of middleware execution?
Order matters ‚úÖ
Registered order = request execution order.
Reverse order = response execution order.
Example:
Request ‚Üí Middleware1 ‚Üí Middleware2 ‚Üí Controller
Response ‚Üê Middleware1 ‚Üê Middleware2 ‚Üê Controller

9. Where do we configure middleware?
In Program.cs (or Startup.cs in older .NET Core versions), inside the app pipeline:

var app = builder.Build();
app.UseRouting();
app.UseAuthorization();

10. Can middleware depend on Dependency Injection?
Yes ‚úÖ You can inject services via constructor:

public class MyMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<MyMiddleware> _logger;
    public MyMiddleware(RequestDelegate next, ILogger<MyMiddleware> logger)
    {
        _next = next; _logger = logger;
    }
    public async Task InvokeAsync(HttpContext context) { ... }
}

11. When should you use Middleware vs Delegating Handler?
‚úî Middleware ‚Üí For cross-cutting concerns at application level (logging, auth, exception handling).
‚úî Delegating Handler ‚Üí For HTTP client pipelines (HttpClientFactory).

12. Give some examples of custom middleware use cases.
‚úî Logging request/response.
‚úî Custom error handling.
‚úî IP restriction / request filtering.
‚úî Adding custom headers.
‚úî Request timing / performance monitoring.

üîπ Caching in .NET Core ‚Äì Interview Q&A

1. What is caching in .NET Core?

Answer: Caching is a technique of storing frequently used data in memory so that future requests can be served faster without hitting the database or external service repeatedly.

2. What are the types of caching in ASP.NET Core?

  • In-Memory Caching ‚Üí Data stored in server memory.
  • Distributed Caching ‚Üí Shared cache across servers (e.g., SQL Server, Redis).
  • Response Caching ‚Üí Caches full HTTP responses.
  • Output Caching (from .NET 7) ‚Üí Caches action results at framework level.

3. What is the difference between In-Memory Cache and Distributed Cache?

Answer:

  • In-Memory Cache ‚Üí Stored in memory of a single server, fast but not shared.
  • Distributed Cache ‚Üí Stored centrally (Redis, SQL Server), available to multiple app servers.

4. How do you configure In-Memory Caching in .NET Core?

builder.Services.AddMemoryCache();
public class ProductController : Controller
{
    private readonly IMemoryCache _cache;
    public ProductController(IMemoryCache cache) => _cache = cache;

    public IActionResult GetProduct()
    {
        var product = _cache.GetOrCreate("product_1", entry =>
        {
            entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
            return "Cached Product";
        });
        return Ok(product);
    }
}

5. What is Sliding vs Absolute Expiration?

Answer:

  • Absolute Expiration ‚Üí Item removed after a fixed time.
  • Sliding Expiration ‚Üí Item removed if not accessed within a given time.

6. How do you implement Distributed Caching?

builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = "localhost:6379";
});
private readonly IDistributedCache _cache;
public MyService(IDistributedCache cache) => _cache = cache;

public async Task<string> GetDataAsync()
{
    var data = await _cache.GetStringAsync("key1");
    if (data == null)
    {
        data = "Hello from DB";
        await _cache.SetStringAsync("key1", data,
            new DistributedCacheEntryOptions
            {
                AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5)
            });
    }
    return data;
}

7. What is Response Caching Middleware?

app.UseResponseCaching();
[ResponseCache(Duration = 60)]
public IActionResult Index() => View();

8. What is the difference between Response Caching and Output Caching?

  • Response Caching ‚Üí Client/proxy caching using HTTP headers.
  • Output Caching (new in .NET 7+) ‚Üí Caches action results at server-side framework level.

9. How do you remove an item from cache?

_cache.Remove("product_1");

10. What is cache invalidation?

Answer: The process of removing or refreshing stale cache data when the underlying source data changes.

  • Time-based (absolute/sliding expiration)
  • Manual invalidation
  • Event-driven (invalidate when DB updates)

11. Can you cache complex objects?

Answer: Yes ‚úÖ In In-Memory Cache directly. For Distributed Cache (Redis/SQL), serialize object (JSON, binary) before storing.

12. What are cache tags or cache dependencies?

Answer: Use IChangeToken to invalidate cache when configs or external triggers change.

13. What are best practices for caching in .NET Core?

  • Use caching for read-heavy, infrequently changing data.
  • Use Distributed Cache in load-balanced apps.
  • Avoid caching sensitive data.
  • Set proper expiration policies.
  • Monitor cache hit/miss ratios.

14. What is cache stampede and how do you prevent it?

Answer: Cache stampede = multiple requests hit the DB when cache expires at the same time.

  • Locking (only one request refreshes cache).
  • Staggered Expiration.
  • Background refresh.

15. What is the difference between Cache vs Session in ASP.NET Core?

  • Cache ‚Üí Stores data for performance, not user-specific.
  • Session ‚Üí Stores user-specific data (e.g., shopping cart).

üîπ Solid principle

1. What are SOLID Principles?

Answer: SOLID is an acronym for five object-oriented design principles that improve code maintainability and flexibility:

  • S ‚Üí Single Responsibility Principle (SRP)
  • O ‚Üí Open/Closed Principle (OCP)
  • L ‚Üí Liskov Substitution Principle (LSP)
  • I ‚Üí Interface Segregation Principle (ISP)
  • D ‚Üí Dependency Inversion Principle (DIP)

2. What is the Single Responsibility Principle (SRP)?

Answer: A class should have only one reason to change. It should have a single responsibility.

public class InvoicePrinter
{
    public void Print(Invoice invoice) { /* print logic */ }
}

Here, InvoicePrinter only handles printing, not calculating totals or storing data.

3. What is the Open/Closed Principle (OCP)?

Answer: Software entities should be open for extension but closed for modification.

public abstract class Shape
{
    public abstract double Area();
}

public class Circle : Shape
{
    public double Radius { get; set; }
    public override double Area() => Math.PI * Radius * Radius;
}

You can add new shapes without modifying existing code.

4. What is the Liskov Substitution Principle (LSP)?

Answer: Derived classes must be substitutable for their base classes without breaking the application.

public class Bird { public virtual void Fly() { } }
public class Ostrich : Bird 
{
    public override void Fly()
    {
        throw new NotSupportedException(); // Violates LSP
    }
}

5. What is the Interface Segregation Principle (ISP)?

Answer: No client should be forced to depend on methods it does not use. Split large interfaces into smaller, specific ones.

public interface IPrinter { void Print(); }
public interface IScanner { void Scan(); }

6. What is the Dependency Inversion Principle (DIP)?

Answer: High-level modules should not depend on low-level modules. Both should depend on abstractions (interfaces).

public interface IMessageService { void SendMessage(string msg); }

public class EmailService : IMessageService
{
    public void SendMessage(string msg) { /* send email */ }
}

public class Notification
{
    private readonly IMessageService _service;
    public Notification(IMessageService service) => _service = service;
    public void Alert(string msg) => _service.SendMessage(msg);
}

7. Why are SOLID principles important in interviews?

Answer: They demonstrate your understanding of maintainable, scalable, and clean OOP design. Interviewers often assess your ability to write flexible and testable code using SOLID principles.

üîπ Design Patterns in .NET

1. What is the Singleton Pattern?

Answer: Ensures a class has only one instance and provides a global point of access to it.

public class Singleton
{
    private static readonly Singleton _instance = new Singleton();
    private Singleton() { }
    public static Singleton Instance => _instance;
}

2. What is the Builder Pattern?

Answer: Separates the construction of a complex object from its representation, allowing the same construction process to create different representations.

public class Car
{
    public string Engine { get; set; }
    public string Wheels { get; set; }
}

public class CarBuilder
{
    private Car _car = new Car();
    public CarBuilder SetEngine(string engine) { _car.Engine = engine; return this; }
    public CarBuilder SetWheels(string wheels) { _car.Wheels = wheels; return this; }
    public Car Build() => _car;
}

3. What is the Prototype Pattern?

Answer: Creates new objects by copying an existing object, reducing the cost of creating new instances.

public class Prototype
{
    public int Id { get; set; }
    public Prototype Clone() => (Prototype)this.MemberwiseClone();
}

4. What is the Factory Pattern?

Answer: Provides an interface for creating objects but allows subclasses to alter the type of objects that will be created.

public interface IShape { void Draw(); }

public class Circle : IShape { public void Draw() => Console.WriteLine("Circle"); }
public class Square : IShape { public void Draw() => Console.WriteLine("Square"); }

public class ShapeFactory
{
    public static IShape GetShape(string type) =>
        type == "Circle" ? new Circle() : new Square();
}

5. What is the Adapter Pattern?

Answer: Allows incompatible interfaces to work together by converting the interface of a class into another interface the client expects.

public interface ITarget { void Request(); }

public class Adaptee { public void SpecificRequest() => Console.WriteLine("Adaptee request"); }

public class Adapter : ITarget
{
    private readonly Adaptee _adaptee = new Adaptee();
    public void Request() => _adaptee.SpecificRequest();
}

6. What is the Facade Pattern?

Answer: Provides a simplified interface to a complex subsystem.

public class SubSystemA { public void OperationA() => Console.WriteLine("A"); }
public class SubSystemB { public void OperationB() => Console.WriteLine("B"); }

public class Facade
{
    private SubSystemA a = new SubSystemA();
    private SubSystemB b = new SubSystemB();
    public void Operation() { a.OperationA(); b.OperationB(); }
}

7. What is the Decorator Pattern?

Answer: Adds behavior to objects dynamically without modifying their structure.

public interface IMessage { void Send(); }

public class EmailMessage : IMessage
{
    public void Send() => Console.WriteLine("Sending Email");
}

public class MessageDecorator : IMessage
{
    private readonly IMessage _message;
    public MessageDecorator(IMessage message) => _message = message;
    public void Send() { Console.WriteLine("Logging..."); _message.Send(); }
}

8. What is the Repository Pattern?

Answer: Separates the data access logic from the business logic by providing a collection-like interface for accessing domain objects.

public interface IProductRepository
{
    IEnumerable GetAll();
}

public class ProductRepository : IProductRepository
{
    private readonly AppDbContext _context;
    public ProductRepository(AppDbContext context) => _context = context;
    public IEnumerable GetAll() => _context.Products.ToList();
}

9. What is the CQRS Pattern?

Answer: Command Query Responsibility Segregation separates read and write operations for better scalability and maintainability.

// Command
public class CreateOrderCommand { public int OrderId { get; set; } }
// Query
public class GetOrderQuery { public int OrderId { get; set; } }
// Handlers would process commands and queries separately.

1. What is a Table in SQL Server?

Answer: A table is a database object used to store structured data in rows and columns.

CREATE TABLE Employees
(
    EmployeeID INT PRIMARY KEY,
    Name NVARCHAR(50),
    Department NVARCHAR(50),
    Salary DECIMAL(10,2)

2. What is a View and when would you use it?

Answer: A view is a virtual table that presents data from one or more tables. Use it to simplify complex queries or restrict access.

CREATE VIEW EmployeeView AS
SELECT Name, Department, Salary
FROM Employees
WHERE Salary > 50000;

3. What are Functions in SQL Server?

Answer: Functions are reusable routines that return a single value (scalar) or table (table-valued). Useful for calculations or transforming data.

-- Scalar Function
CREATE FUNCTION dbo.GetAnnualSalary(@MonthlySalary DECIMAL)
RETURNS DECIMAL
AS
BEGIN
    RETURN @MonthlySalary * 12
END;

4. What are Stored Procedures and why are they used?

Answer: Stored Procedures are precompiled SQL statements that can perform operations like CRUD, complex queries, and transaction management. They improve performance and security.

CREATE PROCEDURE GetEmployeesByDepartment
    @Dept NVARCHAR(50)
AS
BEGIN
    SELECT * FROM Employees
    WHERE Department = @Dept
END;

5. What is a Trigger and when would you use it?

Answer: A trigger is a special stored procedure that automatically executes in response to certain events on a table (INSERT, UPDATE, DELETE).

CREATE TRIGGER trgAfterInsertEmployee
ON Employees
AFTER INSERT
AS
BEGIN
    PRINT 'A new employee record has been added.'
END;

6. Difference between Stored Procedure and Function?

  • Stored Procedure ‚Üí Can perform insert/update/delete, can return multiple result sets, cannot be used in SELECT directly.
  • Function ‚Üí Returns a single value or table, cannot perform modifications on data, can be used in SELECT queries.

7. What are the benefits of using Views?

  • Simplify complex queries for users.
  • Provide a level of security by exposing only specific columns.
  • Make queries reusable.

8. How do you handle errors in Stored Procedures?

CREATE PROCEDURE SampleProcedure
AS
BEGIN
    BEGIN TRY
        -- Your SQL code here
    END TRY
    BEGIN CATCH
        PRINT 'Error occurred: ' + ERROR_MESSAGE();
    END CATCH

9. Can Triggers cause performance issues?

Answer: Yes ‚úÖ Triggers execute automatically, so poorly written triggers on large tables or frequent operations can slow down performance. Always test and optimize triggers.

10. Best practices for writing Stored Procedures and Functions?

  • Keep logic simple and modular.
  • Use parameters to avoid SQL injection.
  • Handle errors with TRY-CATCH blocks.
  • Use SET NOCOUNT ON to reduce unnecessary network traffic.
  • Comment your code for clarity.

üîπ Test-Driven Development (TDD) & Unit Testing in .NET

1. What is Test-Driven Development (TDD)?

Answer: TDD is a software development approach where you write tests before writing the actual code. The workflow is usually: Red ‚Üí Green ‚Üí Refactor.

  • Red: Write a failing test.
  • Green: Write code to make the test pass.
  • Refactor: Clean the code while keeping tests passing.

2. What is xUnit?

Answer: xUnit is a popular unit testing framework for .NET. It allows you to write and run automated tests for your code.

using Xunit;

public class CalculatorTests
{
    [Fact]
    public void Add_ReturnsSum()
    {
        var calc = new Calculator();
        int result = calc.Add(2, 3);
        Assert.Equal(5, result);
    }
}

3. What is Moq?

Answer: Moq is a mocking framework for .NET. It allows you to create mock objects and simulate dependencies in unit tests.

using Moq;
using Xunit;

public class OrderServiceTests
{
    [Fact]
    public void PlaceOrder_CallsRepository()
    {
        var mockRepo = new Mock();
        var service = new OrderService(mockRepo.Object);

        service.PlaceOrder(new Order());

        mockRepo.Verify(r => r.Save(It.IsAny()), Times.Once);
    }
}

4. What is Jasmine?

Answer: Jasmine is a JavaScript testing framework for unit tests, commonly used for front-end testing (Angular, React).

describe('Calculator', () => {
  it('should add two numbers', () => {
    const calc = new Calculator();
    expect(calc.add(2, 3)).toBe(5);
  });
});

5. Benefits of Test-Driven Development?

  • Ensures code correctness and reduces bugs.
  • Improves code design and maintainability.
  • Makes refactoring safer.
  • Provides automated documentation of functionality.

6. Best practices for unit testing in .NET?

  • Write tests for a single behavior per test.
  • Use descriptive names for test methods.
  • Mock external dependencies to isolate the unit.
  • Run tests frequently and automate with CI/CD.
  • Keep tests fast and repeatable.

7. Difference between Unit Test, Integration Test, and Functional Test?

  • Unit Test: Tests a single class or method in isolation.
  • Integration Test: Tests how multiple components work together.
  • Functional Test: Tests the system from the user's perspective, usually end-to-end.

üîπ SDLC, Agile SCRUM & Azure DevOps CI/CD

1. What is SDLC?

Answer: SDLC (Software Development Life Cycle) is a systematic process for planning, creating, testing, and deploying software. It ensures quality and efficiency at each phase.

  • Planning
  • Requirements Analysis
  • Design
  • Development
  • Testing
  • Deployment
  • Maintenance

2. What is Agile SCRUM methodology?

Answer: Agile SCRUM is an iterative and incremental methodology for managing software projects. Work is divided into sprints, usually 2-4 weeks long, with continuous delivery of working software.

  • Roles: Product Owner, Scrum Master, Development Team
  • Artifacts: Product Backlog, Sprint Backlog, Increment
  • Events: Sprint Planning, Daily Standup, Sprint Review, Sprint Retrospective

3. How do you manage projects end-to-end?

Answer: By following SDLC phases and Agile SCRUM practices:

  • Plan: Gather requirements and define scope.
  • Design: Create architecture, wireframes, and database models.
  • Develop: Write code, implement features, perform unit testing.
  • Test: Execute integration, functional, and regression testing.
  • Deploy: Release to production using CI/CD pipelines.
  • Maintain: Monitor, fix bugs, and implement enhancements.
